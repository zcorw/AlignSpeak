# AlignSpeak Web V2 技术决策文档

本文档记录 web_v2 项目开发前确认的所有技术细节和架构决策。

---

## 1. 架构与路由策略

### 1.1 导航模式
- **决策**：采用无底部 Tab 的强引导流程
- **理由**：符合 GUIDED_FLOW_V2_FINAL.md 的设计，简化用户操作路径
- **"我的"页面访问**：通过顶部右上角入口按钮进入
- **路由方案**：使用 React Router 进行页面导航

### 1.2 页面流转机制
- **页面切换动画**：fade-up 效果（参考 previews/styles.css）
- **路由模式**：使用 `push` 方式（保留浏览器返回能力）
- **全屏遮罩**：使用 Modal 组件（录音专注模式、文本编辑器等场景）

---

## 2. 状态管理

### 2.1 技术选型
- **方案**：Zustand
- **理由**：
  - 轻量（~1KB）
  - TypeScript 类型安全
  - API 简洁，学习成本低
  - 性能优秀（基于订阅机制）
  - 支持 Redux DevTools

### 2.2 全局状态范围
需要全局管理的状态包括：
- 用户认证状态（access_token, refresh_token）
- 当前文章上下文（article_id, segment_id, level）
- 流程恢复状态（current_step, flow_version）
- 未同步数据队列

---

## 3. API 调用与数据层

### 3.1 接口策略
- **主要方式**：调用 BFF 聚合接口（`/api/bff/v1/*`）
- **后端同步**：需要同步开发 BFF 层
- **字段风格**：后端 snake_case，前端接收后转为 camelCase

### 3.2 文章创建流程
- **分段逻辑**：由后端处理（前端提交完整文本）
- **语种检测**：后端自动检测，前端展示结果并允许手动修改

### 3.3 练习录音提交
- **录音格式**：PCM 原始音频数据
- **上传方式**：分片上传（FormData multipart）
- **识别方式**：异步处理 + 轮询检查状态
- **技术实现**：
  - 使用 AudioContext + AudioWorklet 获取 PCM 数据
  - 实时分片上传（建议每 2 秒一片）
  - 后端接收后拼接并编码为 wav

---

## 4. 录音功能

### 4.1 技术方案
- **音频格式**：PCM 分片上传
- **浏览器兼容**：通过 PCM 方案解决跨浏览器编码差异（包括 iOS Safari）
- **实现方式**：AudioContext + AudioWorklet（或降级 ScriptProcessorNode）

### 4.2 限制与约束
- **时长限制**：10 分钟
- **文件大小**：根据采样率计算（约 1 分钟 5-10MB）
- **权限处理**：被拒绝时提示用户无法使用录音功能，不提供替代方案

### 4.3 iOS Safari 兼容性说明
- iOS Safari 不支持 webm 格式
- 通过 PCM 方案统一处理，避免格式兼容问题
- 后端统一编码为 wav 格式

---

## 5. 遮挡渲染

### 5.1 数据结构
- **后端返回**：完整 token 列表，每个 token 带 `masked: true/false` 标记
- **示例**：
```json
{
  "tokens": [
    { "surface": "私", "yomi": "ワタシ", "masked": false },
    { "surface": "は", "yomi": "ハ", "masked": true },
    { "surface": "学生", "yomi": "ガクセイ", "masked": false }
  ]
}
```

### 5.2 视觉效果
- **遮挡样式**：CSS `filter: blur(4px)` 模糊处理
- **偷看功能**：支持长按查看遮挡内容
- **交互反馈**：长按时移除 blur 效果，释放后恢复

---

## 6. 对齐结果可视化

### 6.1 数据结构
- **方案**：B 方案（分离展示原文和识别文本）
- **数据格式**：
```json
{
  "ref_tokens": [
    { "surface": "私", "yomi": "ワタシ" },
    { "surface": "も", "yomi": "モ" },
    { "surface": "学生", "yomi": "ガクセイ" }
  ],
  "hyp_tokens": [
    { "surface": "私", "yomi": "ワタシ" },
    { "surface": "は", "yomi": "ハ" }
  ],
  "alignment": [
    { "ref_idx": 0, "hyp_idx": 0, "status": "correct" },
    { "ref_idx": 1, "hyp_idx": 1, "status": "substitute" },
    { "ref_idx": 2, "hyp_idx": null, "status": "missing" }
  ],
  "noise_spans": [
    { "start": 2, "end": 4, "reason": "repeat" }
  ]
}
```

### 6.2 展示方式
- **布局**：上下两行对比显示
  - 上行：原文（ref_tokens）
  - 下行：识别文本（hyp_tokens）
- **状态标记**：
  - `correct`：绿色
  - `substitute`：红色删除线
  - `missing`：黄色背景（原文中标记）
  - `insert`：灰色（识别文本中标记）

### 6.3 噪声块处理
- **可折叠**：支持展开/收起
- **视觉标识**：灰色背景 + "重复片段"提示文字
- **默认状态**：折叠显示
- **交互详情**：不支持点击 token 查看详细信息（读音、词性等）

## 8. 同步失败处理

### 8.1 未同步数据存储
- **存储位置**：localStorage
- **存储结构**：
```json
{
  "pendingActions": [
    {
      "id": "uuid",
      "type": "practice_attempt",
      "payload": { "articleId": "...", "segmentId": "...", "audioBlob": "..." },
      "createdAt": "2026-03-05T10:30:00Z",
      "retryCount": 0,
      "lastError": null
    }
  ]
}
```

### 8.2 后台重试策略
- **重试间隔**：指数退避（1s, 2s, 4s, 8s, 16s, 32s）
- **最大重试次数**：6 次
- **过期清理**：24 小时后自动清理未成功的数据

### 8.3 用户手动重试
- **触发方式**：顶部显示未同步徽章，点击后展示列表
- **UI 反馈**：
  - 重试中：显示 loading 状态
  - 重试成功：从列表移除，显示成功提示
  - 重试失败：显示错误信息，允许再次重试或删除

## 9. UI 组件库与样式

### 9.1 组件库选型
- **方案**：Material-UI (MUI)
- **理由**：
  - 成熟稳定，社区活跃
  - TypeScript 支持完善
  - 组件丰富，覆盖大部分需求
  - 自定义主题能力强

### 9.2 样式实现
- **基准**：参考 previews 样式（深色主题、圆角、渐变等）
- **灵活度**：允许根据 MUI 框架特色进行调整，不需要 100% 复刻
- **主题切换**：MVP 阶段不实现，仅支持暗色主题

---

## 10. 多语言支持

### 10.1 前端 UI 国际化
- **方案**：支持中英文切换
- **实现**：使用 i18n 库（如 react-i18next）
- **默认语言**：根据浏览器语言自动选择，默认中文

### 10.2 文章语种支持
- **支持语种**：日语、中文、英语
- **日语注音**：
  - 主要在录音专注遮罩处显示
  - 使用 HTML `<ruby>` 标签渲染
  - 数据来源：后端返回的 `yomi` 字段
- **特殊处理**：暂不支持 RTL 语言（如阿拉伯语）

### 10.3 语种检测失败处理
- **策略**：强制用户手动选择语种后才能继续
- **理由**：语种直接影响分词、TTS、STT 等核心功能，不能使用错误的默认值

## 11. 性能优化

### 11.1 虚拟滚动
- **决策**：MVP 阶段不实现虚拟滚动
- **理由**：
  - 文章段落限制在 30 段以内
  - 单段 token 数量通常在几十到几百之间
  - 现代浏览器渲染性能足够
  - 虚拟滚动会增加复杂度，且与 token 级交互可能产生冲突

### 11.2 对齐结果展示
- **决策**：不分页，一次性展示全部对齐结果
- **理由**：
  - 对齐结果是按段落练习的，单段内容有限
  - 用户需要完整查看对比原文和识别结果
  - 分页会打断阅读连贯性
- **优化**：通过"展开/收起噪声块"来减少视觉负担

### 11.3 音频预加载
- **决策**：点击播放时才加载，不预加载
- **理由**：
  - TTS 音频文件较大，预加载会浪费带宽
  - 后端已有 TTS 缓存机制，二次请求速度快
  - MVP 阶段优先保证功能完整性
- **缓存策略**：依赖浏览器 HTTP 缓存，不实现前端音频缓存（IndexedDB）

## 12. 测试策略

### 12.1 测试框架选型
- **方案**：Vitest + React Testing Library
- **理由**：
  - Vitest 与 Vite 无缝集成，配置简单
  - 启动速度快，支持 HMR
  - API 与 Jest 兼容，学习成本低
  - React Testing Library 是 React 测试的事实标准

### 12.2 测试覆盖率要求
- **核心业务逻辑**：必须有单元测试
  - Zustand store（状态管理逻辑）
  - 工具函数（如 token 处理、时间格式化等）
  - 关键算法（如对齐结果处理、遮挡计算等）
  - 目标覆盖率：80%+
- **组件测试**：选择性覆盖
  - 复杂交互组件（如录音控制、遮罩渲染）需要测试
  - 简单展示组件可以不测试
  - 目标覆盖率：整体 60%+
- **E2E 测试**：MVP 阶段不实现
  - 理由：E2E 测试维护成本高，MVP 阶段优先保证功能完整性
  - 后续可以考虑使用 Playwright 覆盖关键流程

### 12.3 Mock 数据维护
- **存放位置**：`src/__mocks__/fixtures/` 目录
- **组织方式**：按模块分类（如 `articles.ts`, `practice.ts`, `alignment.ts`）
- **契约同步**：
  - Mock 数据结构必须与 BFF 契约保持一致
  - 在 `fixtures/` 中维护 TypeScript 类型定义
  - 后端 API 变更时同步更新 Mock 数据
- **时效性保证**：
  - 在 CI 中运行类型检查，确保 Mock 数据类型正确
  - 定期与后端对齐接口变更
