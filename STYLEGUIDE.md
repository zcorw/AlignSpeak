# AlignSpeak 代码风格指南

## 1. 通用原则
- 可读性优先于技巧性写法。
- 函数保持短小、单一职责。
- 避免隐式副作用，数据流要显式。
- 优先组合而非深层继承。

## 2. 强制工程约束（必须遵守）
- 分层架构：必须按 `UI层 -> 应用层 -> 领域层 -> 基础设施层` 分层。
- 单向依赖：上层可依赖下层，下层禁止反向依赖上层。
- 职责隔离：一个模块只承担一种核心职责，不混合 UI/业务/持久化逻辑。
- 可测试优先：新增业务逻辑必须可单测，不可测试代码视为设计缺陷。
- 扩展优先：优先接口与策略扩展，避免修改核心流程分支堆叠。
- 禁止万能工具类：禁止引入承载多域逻辑的 `utils`/`helper` 巨石类。
- 禁止控制层写业务逻辑：路由/控制器仅做参数校验、鉴权、编排，不写核心业务决策。
- 禁止组件直接发请求：UI 组件不得直接调用 HTTP，统一通过应用层 action/service。
- 禁止隐式耦合：禁止依赖全局隐式状态、初始化顺序、副作用时序。

## 3. 前端风格（React + TypeScript）
- 使用严格类型，非必要不使用 `any`。
- 状态就近管理，跨页面共享才上升到全局状态。
- 组件按职责拆分：输入、录音、遮挡渲染、结果高亮、报告面板。
- 异步交互必须具备：加载态、错误态、可重试路径。
- 展示型组件尽量无副作用。
- 组件内禁止直接请求接口，改由 `hooks/useCase` 调用应用层。

## 4. 后端风格（FastAPI + Python）
- 公共函数和服务接口必须带类型标注。
- 请求/响应结构统一使用 Pydantic 模型。
- 路由层保持薄，核心逻辑下沉到 service 层。
- STT/TTS 供应商接入通过适配器隔离，不在业务逻辑中硬编码。
- 异常映射为稳定错误码，不直接向外抛底层异常文本。
- 鉴权逻辑集中在 `Auth Service` 与中间件/依赖层，不分散在业务 handler。
- 路由/控制层禁止写业务规则判断。

## 5. API 风格
- 字段统一 `snake_case`。
- 响应结构稳定，避免无版本直接破坏兼容。
- 错误响应统一 `error.code/error.message/error.details`。
- 示例请求和返回应与真实 schema 保持一致。
- 受保护接口统一使用 Bearer Token，并在 OpenAPI 中显式标注安全方案。

## 6. 账号与安全代码风格
- 密码处理必须使用成熟哈希算法（`Argon2id`/`bcrypt`），禁止明文落库。
- Refresh Token 仅保存 hash，比较时走恒时比较逻辑。
- 刷新流程必须执行 token 轮换，避免旧 token 重放。
- 所有按用户隔离的查询必须显式带 `user_id` 过滤条件。
- 禁止在日志中输出密码、token、完整敏感 payload。

## 7. 对齐与遮挡代码风格
- 对齐逻辑必须分层：
  - 规范化
  - 分词
  - DP 匹配
  - 后处理（`noise_spans`）
- 规则遮挡在相同输入下必须可复现（确定性）。
- 对齐模式（`yomi`/`surface`）和评分参数必须入库记录。

## 8. 命名风格
- 使用语义明确的英文命名。
- 布尔命名使用谓词形式：`is_cached`、`has_noise`、`should_mask`。
- 避免歧义缩写，优先完整词。

## 9. 注释风格
- Python 函数使用 docstring；TypeScript/Vue 函数使用 JSDoc 或等价块注释。
- 注释解释“为什么”，不是“做了什么”。
- 仅在意图不直观时加注释。
- TODO 必须可执行，建议附 issue 或负责人。

## 10. 测试风格
- 单测重点覆盖：规范化、对齐状态、遮挡计划。
- 契约测试重点覆盖：API 字段与枚举稳定性。
- 端到端测试至少覆盖一条完整训练闭环。
- 测试命名描述行为，不描述实现细节。
- 账号相关至少覆盖：
  - 注册/登录成功与失败
  - token 过期与刷新
  - 刷新 token 轮换与撤销
  - 跨用户资源越权拦截
- 控制层测试与业务层测试分离，业务规则覆盖必须在 service/usecase 层完成。

## 11. UI 与文案风格
- 对齐状态颜色语义固定：
  - `correct`：正向色
  - `substitute/missing/insert`：警示色
  - `noise`：中性色并支持折叠提示
- 用户文案保持短句、可操作、避免歧义。
- 多语言场景下避免在同一短标签中混排过多脚本。

## 12. 格式化与静态检查
- 前后端都必须接入 formatter + linter。
- CI 中 lint/type-check/test 失败默认不可合并（初始化阶段特殊豁免除外）。
- 需增加依赖方向校验（如 lint rule / boundary rule），阻止反向依赖。

## 13. 文档风格
- 文档要可执行、可验证，避免空泛描述。
- 行为变更时同步更新：
  - 架构边界
  - API 契约
  - 状态枚举与约定
- 优先给出贴近 MVP 的真实示例。
